# Contributing

Thank you for your interest in contributing to this project.

## Ground rules

- Follow the Code of Conduct.
- Keep contributions focused and reviewable.
- Prefer explicitness over implicit behavior.
- Avoid introducing unnecessary dependencies or abstractions.

## What to submit

### Bug reports

Please include:
- Clear steps to reproduce
- Expected vs actual behavior
- Relevant logs or error output
- Environment details (OS, version, commit)

### Feature requests

Please describe:
- The problem being solved
- The proposed solution
- Alternatives considered
- Potential impact on security or compatibility

### Security issues

Do **not** open public issues for suspected vulnerabilities.

Use the private security advisory mechanism as described in `SECURITY.md`.

## Development workflow

1. Fork the repository
2. Create a scoped branch (`feature/<name>` or `fix/<name>`)
3. Keep commits small and descriptive
4. Open a pull request with context and rationale

## Verification

Contributors are responsible for verifying their changes.

If automated tests are not available, include a brief description of manual verification performed.
